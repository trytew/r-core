流程简述

1. 初始化内存空间
   
   1. 先由`mm:init_heap`创建堆空间分配器`HEAP_ALLOCATOR`，设置内核堆空间，大小为`3MB`
   
   2. 通过静态变量的初始化创建全局物理页帧管理器`FRAME_ALLOCATOR`，因为还在实模式（S-Mode）下，因此`FrameAllocator.recycled`字段的`Vec`使用的堆内存在由`mm:init_heap`创建堆空间中
   
   3. 通过静态变量的初始化实例化内核空间`KERNEL_SPACE`，此时物理内存分布如下：
      
      ```shell
      低地址  
      |                                           |  
      |                                           |  
      |-------------------------------------------|--> 内核空间起始  
      | .text   内核代码                            |  
      |   |                                       |  
      |   |--trampoline                           |  
      |                                           |  
      | .rodata 常量                               |  
      | .data   已初始化全局变量                     |  
      | .bss    零初始化全局变量                     |  
      |   |                                       |  
      |   |-- HEAP_SPACE [3MB]（内核堆）            |  
      |   |    |                                  |  
      |   |    |  (KERNEL_SPACE.areas)            |  
      |   |    |--Vec<MapArea> 的元素              |  
      |   |    |   |                              |  
      |   |    |   |--MapArea                     |  
      |   |    |       |                          |  
      |   |    |       |--vpn_range               |  
      |   |    |       |                          |  
      |   |    |       |--data_frames (BTreeMap)  |  
      |   |    |       |   |                      |  
      |   |    |       |   |-- BTreeMap Item      |  
      |   |    |       |                          |  
      |   |    |       |--map_type                |  
      |   |    |       |                          |  
      |   |    |       |--map_perm                |  
      |   |    |                                  |  
      |   |    |  (KERNEL_SPACE.PageTable.frames) |  
      |   |    |--Vec<FrameTracker> 的元素         |  
      |   |    |   |                              |  
      |   |    |   |--FrameTracker                |  
      |   |                                       |  
      |   |-- FRAME_ALLOCATOR                     |  
      |   |                                       |  
      |   |-- KERNEL_SPACE                        |  
      |        |                                  |  
      |        |-- PageTable                      |  
      |        |    |                             |  
      |        |    |-- root_ppn                  |  
      |        |    |                             |  
      |        |    |-- frames (Vec)              |  
      |        |                                  |  
      |        |-- areas (Vec<MapArea>)           |  
      |                                           |  
      | ekernel                                   |  
      |-------------------------------------------|--> 内核内存起始  
      | | 一级页表页（4KB）                          |  
      | ----------------------------------------  |  
      | | 内存页（4KB）                             |  
      | ----------------------------------------  |  
      | | 内存页（4KB）                             |  
      | ----------------------------------------  |  
      | | ...                                     |  
      |                                           |  
      |（内核内存和内核堆一共占用 128MB 物理内存）      |  
      |（内核内存是应用程序和内核都一起共用的内存空间）   |  
      | MEMORY_END                                |  
      |-------------------------------------------|--> 内核内存结束  
      |                                           |  
      高地址  
      ```
      
      `MemorySet`是一个内存区域存放集合，记录该集合的所有页表帧信息和内存区域信息集合
      
      1. 创建内核空间集合`KERNEL_SPACE`，它存在于内核堆`HEAP_ALLOCATOR`中，其中包括了所有用户程序的可使用内存空间信息
      
      2. 将跳板映射到最高虚拟内存地址
      
      3. 根据内核代码每个段的标识映射到内存中，因为采用的是恒等映射方式，因此标识的地址就是虚拟内存地址。但映射的过程中仍需要走虚拟内存的映射流程，因为开启MMU后会使用虚拟地址的方式对内容进行寻址
   
   4. 开启`MMU`，开启`MMU`后代码的所有内存寻址都将使用虚拟内存，因为我们的内核代码区域映射使用的是恒等映射方式，因此执行`satp::write(satp)`开启`MMU`后后续代码依旧能正常运行

2. 设置内核`Trap`陷入的处理方法，因为目前不会再向上一级“陷入”，因此如果内核触发了“陷入”则直接返回`panic`错误

3. 设置时间中断并开启

4. 通过静态变量的初始化创建任务管理器`TASK_MANAGER`，它存在于内核堆`HEAP_ALLOCATOR`中，它管理了所有用户态程序
   
   1. 读取用户态程序内容，它的内容由`global_asm!(include_str!("./linker_app.asm"))`语句在内核启动时加载
   
   2. 创建新的页表，内核和每个用户态程序各自拥有自己的页表
   
   3. 将跳板映射到最高虚拟内存地址，这个时候所有用户态程序和内核的跳板虚拟地址地址都是同一个
   
   4. 因为用户态的程序使用的是正常的`elf`编译格式，因此读取`elf`文件信息并按它的协议格式将程序映射进虚拟内存，且每个内存页都需要加上`U`权限使得CPU在特权级时也能访问这些内存的内容。在`elf`信息中记录的内存地址均为虚拟内存地址
      
      ```shell
      elf 头表项含义:
          将代码编译成 elf 之后会生成一个程序头表，按程序中是否需要加载进内存、所在虚拟地址、是否能执行、读写、占用内存大小等信息将单个或多个 段 合并记录成多个头表项，每个项代表的信息都不一致，如图:
      
          Segment (PT_LOAD #0)
          |--.text
          |--.rodata
          |--.eh_frame
          |
          Segment (PT_LOAD #1)
          |--.data
          |--.bss  
      ```
      
      每个用户态程序均有以下内容
      
      1. 用户态程序内容（包含`text段`、`data`段等等）
      
      2. 灰页
      
      3. 用户栈
      
      4. 用户堆空间
      
      5. 内核栈
      
      6. 堆空间
      
      7. “陷入”上下文（仅次于跳板页）
      
      8. 跳板页（最高内存页）
   
   5. 实例化用户态程序的控制块`task_control_block`，其中应用上下文`task_cx`中的`ra`被设置为“陷入”函数`trap_return`的起始地址，代表在`__switch`函数执行完`ret`后就会跳转到`trap_return`函数。这是从内核态切换到用户态的关键，因为`trap_return`函数以`sret`结束，执行它的时候会将内核态切换成用户态并跳转到`spec`寄存器记录的内存地址继续执行指令

5. 运行第一个用户态程序
   
   1. 创建一个空白的应用上下文
   
   2. 调用`__switch`汇编函数运行第一个用户态应用，`ret`执行后跳转到`trap_return`函数执行（尚在内核态）
   
   3. “陷入”处理函数执行（尚在内核态）
      
      1. 设置当前应用的“陷入”处理函数地址，即跳板地址，因为所有用户态的程序以及内核中“陷入”处理函数地址都是同一个虚拟地址（即所以用户态程序以及内核都有一份“陷入”处理函数的复印件），因此无需转换内核和用户态程序的虚拟地址
      
      2.
