内核和应用地址空间

第五章第三节

os:

```shell
cd os
./build.sh
./run.sh
```

user:

```shell
cd user
./build.sh
./run.sh
```

流程简述

1. 点亮CPU
   
   ```shell
   QEMU
     ↓
   rustsbi-qemu.bin  (M-mode)
     ↓  mret
   _start (entry.asm)
     ↓
   设置 sp = boot_stack_top
     ↓
   call rust_main
   ```
   
   1. `rustsbi-qemu.bin`是一个`boot-loader`文件，CPU 上电后默认存在`M-mode`特权级。这个文件主要的作用是：
      
      - 释义1：
        
        1. 初始化基本硬件
        
        2. 设置好特权级转换
        
        3. 提供 SBI 接口
        
        4. 设置`S-mode`模式，即`mret`执行后切换的 CPU 模式
        
        5. 执行`mret`跳转到`_start`执行
      
      - 释义2：
        
        1. 设置 mtvec
        
        2. 配置中断
        
        3. 设置 PMP（物理内存保护）
        
        4. 允许 S-mode 访问内存
        
        5. 配置 timer
        
        6. 提供 SBI 调用处理函数
      
      以上两个释义是同一个意思不同的说法，方便理解
   
   2. 在`entry.asm`会初始化一个内核栈，在切换到其他用户态程序之前所有栈的使用都是它，是一个非常关键的地方
   
   > PC 的 CPU 点亮流程（Bootloader）与`RISC-V`不太一样，通常 PC 的 Bootloader 流程会分为两个阶段：
   > 
   > 1. 阶段1：
   >    
   >    1. 上电
   >    
   >    2. BIOS初始化硬件
   >    
   >    3. BIOS在启动设备（硬盘 / U盘）读取 **第一个扇区（512字节）**
   >    
   >    4. 检查最后两个字节是否为`0x55AA`，如果不是，就认为这个设备不可启动，这个扇区成为`MBR（Master Boot Record）`
   >    
   >    5. BIOS会把这 **512字节加载到内存**的`0x7C00`位置，然后跳转执行
   >    
   >    6. 切换CPU状态（有时）
   >    
   >    7. 读取磁盘
   >    
   >    8. 把阶段2的`Bootloader`程序读进内存
   >    
   >    9. 跳转执行
   > 
   > 2. 阶段2：
   >    
   >    1. 识别文件系统（FAT / ext）
   >    
   >    2. 读取内核文件
   >    
   >    3. 加载到内存
   >    
   >    4. 建立初始内存环境
   >    
   >    5. 跳转到内核入口

2. 初始化内存空间
   
   1. 先由`mm:init_heap`创建堆空间分配器`HEAP_ALLOCATOR`，设置内核堆空间，大小为`3MB`
   
   2. 通过静态变量的初始化创建全局物理页帧管理器`FRAME_ALLOCATOR`，它代表那些页帧已被拥有，防止数据被覆盖，因为还在`S-Mode`下，因此`FrameAllocator.recycled`字段的`Vec`使用的堆内存在由`mm:init_heap`创建堆空间中
   
   3. 通过静态变量的初始化实例化内核空间`KERNEL_SPACE`，此时物理内存分布如下：
      
      ```shell
      低地址  
      |                                           |  
      |                                           |  
      |-------------------------------------------|--> 内核空间起始  
      | .text   内核代码                            |
      |   |                                       |
      |   |--trampoline                           |
      |                                           |
      | .rodata 常量                               |
      | .data   已初始化全局变量                     |
      | .bss    零初始化全局变量                     |
      |   |                                       |
      |   |-- HEAP_SPACE [3MB]（内核堆）            |
      |   |    |                                  |
      |   |    |  (KERNEL_SPACE.areas)            |
      |   |    |--Vec<MapArea> 的元素              |
      |   |    |   |                              |
      |   |    |   |--MapArea                     |
      |   |    |       |                          |
      |   |    |       |--vpn_range               |
      |   |    |       |                          |
      |   |    |       |--data_frames (BTreeMap)  |
      |   |    |       |   |                      |
      |   |    |       |   |-- BTreeMap Item      |
      |   |    |       |                          |
      |   |    |       |--map_type                |
      |   |    |       |                          |
      |   |    |       |--map_perm                |
      |   |    |                                  |
      |   |    |  (KERNEL_SPACE.PageTable.frames) |
      |   |    |--Vec<FrameTracker> 的元素         |
      |   |    |   |                              |
      |   |    |   |--FrameTracker                |
      |   |    |                                  |
      |   |    |--INITPROC 与堆分配的相关内容        |
      |   |    |                                  |
      |   |    |--TASK_MANAGER 与堆分配的相关内容    |
      |   |                                       |
      |   |-- FRAME_ALLOCATOR                     |
      |   |                                       |
      |   |-- KERNEL_SPACE                        |
      |   |    |                                  |
      |   |    |-- PageTable                      |
      |   |    |    |                             |
      |   |    |    |-- root_ppn                  |
      |   |    |    |                             |
      |   |    |    |-- frames (Vec)              |
      |   |    |                                  |
      |   |    |-- areas (Vec<MapArea>)           |
      |   |                                       |
      |   |-- INITPROC                            |
      |   |                                       |
      |   |-- TASK_MANAGER                        |
      |                                           |
      | ekernel                                   |
      |-------------------------------------------|--> 内核内存起始
      | | 一级页表页（4KB）                          |
      | ----------------------------------------  |
      | | 内存页（4KB）                             |
      | ----------------------------------------  |
      | | 内存页（4KB）                             |
      | ----------------------------------------  |
      | | ...                                     |
      |                                           |
      |（内核内存和内核堆一共占用 128MB 物理内存）      |
      |（内核内存是应用程序和内核都一起共用的内存空间）   |
      | MEMORY_END                                |
      |-------------------------------------------|--> 内核内存结束
      |                                           |
      高地址
      ```
      
      `MemorySet`是一个内存区域存放集合，记录该集合的所有页表帧信息和内存区域信息集合
      
      1. 创建内核空间集合`KERNEL_SPACE`，它存在于内核堆`HEAP_ALLOCATOR`中，`KERNEL_SPACE`映射了所有物理内存空间，而且它映射内存空间时并不调用`FRAME_ALLOCATOR`，因此`KERNEL_SPACE`可以和应用程序映射的内存空间重叠，即在内核虚拟页表状态下在`KERNEL_SPACE`中可以找到所有已使用的物理内存内容
      
      2. 将跳板映射到最高虚拟内存地址
      
      3. 根据内核代码每个段的标识映射到内存中，因为采用的是恒等映射方式，因此标识的地址就是虚拟内存地址。但映射的过程中仍需要走虚拟内存的映射流程，因为开启MMU后会使用虚拟地址的方式对内容进行寻址
   
   4. 开启`MMU`，开启`MMU`后代码的所有内存寻址都将使用虚拟内存，因为我们的内核代码区域映射使用的是恒等映射方式，因此执行`satp::write(satp)`开启`MMU`后后续代码依旧能正常运行

3. 加载初始应用的内容到内存
   
   1. 读取应用程序内容，它的内容由`global_asm!(include_str!("./linker_app.asm"))`语句在内核启动时加载
   
   2. 创建新的页表，内核和每个应用程序各自拥有自己的页表
   
   3. 将跳板映射到最高虚拟内存地址，这个时候所有应用程序和内核的跳板虚拟地址地址都是同一个
   
   4. 因为用户态的程序使用的是正常的`elf`编译格式，因此读取`elf`文件信息并按它的协议格式将程序映射进虚拟内存，且每个内存页都需要加上`U`权限使得CPU在特权级时也能访问这些内存的内容。在`elf`信息中记录的内存地址均为虚拟内存地址
      
      ```
      elf 头表项含义:
          将代码编译成 elf 之后会生成一个程序头表，按程序中是否需要加载进内存、所在虚拟地址、是否能执行、读写、占用内存大小等信息将单个或多个 段 合并记录成多个头表项，每个项代表的信息都不一致，如图:
      
          Segment (PT_LOAD #0)
          |--.text
          |--.rodata
          |--.eh_frame
          |
          Segment (PT_LOAD #1)
          |--.data
          |--.bss  
      ```
      
      每个应用程序均有以下内容
      
      1. 应用程序内容（包含`text段`、`data`段等等）
      
      2. 灰页
      
      3. 用户栈
      
      4. 用户堆空间
      
      5. 堆空间
      
      6. “陷入”上下文（仅次于跳板页）
      
      7. 跳板页（最高内存页）
   
   5. 在内核虚拟页表中为每个应用创建内核栈（以页帧方式创建，非恒等映射，代表页帧所有权已被拥有，防止被映射导致数据被覆盖），它们的位置仅次于内核的跳板，按应用id以此创建
   
   6. 实例化应用程序的控制块`task_control_block`，其中应用上下文`task_cx`中的`ra`被设置为“陷入”函数`trap_return`的起始地址，代表在`__switch`函数执行完`ret`后就会跳转到`trap_return`函数。这是从内核态切换到用户态的关键，因为`trap_return`函数以`sret`结束，执行它的时候会将内核态切换成用户态并跳转到`spec`寄存器记录的内存地址继续执行指令，每个`task_control_block`都包括以下内容：
      
      1. 应用状态
      
      2. 应用上下文
      
      3. 应用内存区域
      
      4. 应用“陷入”上下文的物理地址（方便在内核虚拟页表的状态下调用）

4. 创建进程管理器`TASK_MANAGER`，将初始应用添加到待运行列表

5. 设置内核`Trap`陷入的处理方法，因为目前不会再向上一级“陷入”，因此如果内核触发了“陷入”则直接返回`panic`错误

6. 设置时间中断并开启

7. 索引所有应用序号和对应的应用名称

8. 打印所有应用

9. 创建进程处理器`PROCESSOR`，并添加一个空白进程上下文作为 **idle 进程上下文**（重点）

10. 运行第一个应用程序
    
    1. 从进程管理器`TASK_MANAGER`中查找一个待运行进程
    
    2. 调用`__switch`汇编函数运行初始进程，`ret`执行后跳转到`trap_return`函数执行（尚在内核态）
    
    3. `trap_return`函数执行（尚在内核态）
       
       1. 设置当前应用的“陷入”上下文。因为每个应用程序的“陷入”上下文虚拟地址都一样，因此只要切换到应用的用户空间并使用`TRAP_CONTEXT`地址即可获取应用的“陷入”上下文
       
       2. 计算`__restore`函数位置。因为每个应用包括内核都存有一份跳板，即“陷入”处理函数的复印件且地址一样，因此只要计算`__restore`函数所在的虚拟地址即可
       
       3. 调用`__restore`汇编函数
          
          1. 先将当前虚拟地址页表切换成需要恢复执行的应用的虚拟地址页表，获取进程正确的“陷入”上下文地址
          
          2. 将需要恢复运行的进程的“陷入”上下文存入`sscratch`，这对应用后续触发“陷入”时有关键作用
          
          3. 保存内核栈`sp`
          
          4. 将栈顶`sp`切换到“陷入”上下文
          
          5. 恢复应用的各个寄存器以及设置`sepc`寄存器，当`sret`调用后会跳转到`spec`记录的地址执行指令，每个应用“陷入”上下文的`sepc`字段记录的都是应用的入口地址
          
          6. 将`sp`切换到进程的用户栈

11. 触发定时器切换进程/触发陷入
    
    无论是切换应用还是触发陷入都会先执行`__alltraps`汇编函数：
    
    1. 因为在调用`__restore`汇编函数时已经将当前应用的“陷入”上下文`TrapContext`存入`sscratch`，因此可以直接交换出来切换成`sp`栈顶，`sscratch`保存当前应用的用户栈栈顶
    
    2. 将当前应用的所有寄存器存入当前应用的“陷入”上下文
    
    3. 将`sstatus`、`sepc`、`sscratch`寄存器的值保存到当前应用的“陷入”上下文以便后续恢复，而`sepc`保存的值则是“陷入”结束（`sret`被调用）后下一条运行的指令，非常重要，关系到应用“陷入”结束后的正常运行
    
    4. 将当前应用的“陷入”上下文中保存的内核虚拟地址页表设置`stap`、“陷入”处理函数地址、内核栈栈顶加载出来
    
    5. 切换到内核空间
    
    6. 跳转“陷入”处理函数执行
    
    7. 执行“陷入”
       
       - 系统调用（中断）
         
         在执行系统调用的分支中，因为`TrapContext`中记录了每个应用的“陷入”上下文物理地址，而内核虚拟地址页表中又恒等映射过所有物理内存地址，因此可以直接查找“陷入”上下文的内容，在系统调用结束后会执行`trap_return`函数恢复应用的用户态状态
       
       - 进程切换
         
         涉及进程切换时会先修改`PROCESSOR`中记录的当前进程的运行状态为待运行，然后将当前应用添加到待运行队列中。
         
         进入`schedule`函数的时候会获取最开始运行在内核中的`idle`进程，它是一个空白进程，但是在最开始调用`__switch`的时候它的进程上下文中的`ra`记录了`__switch`执行完后跳转运行的地址，即`run_tasks()`函数中`loop`的下一次循环，因此当在`schedule`函数中使用`__switch`切换成`idle`进程后，代码会从下一次循环开始执行，即从待运行进程队列中获取下一个运行的进程
         
         获取空闲进程后，使用`__switch`函数切换进程，又因为每个用户进程的`ra`都被设置为`trap_return`，因此在`__switch`结束（调用`ret`）后就会跳转到`trap_return`开始恢复新应用的执行状态。其中每个用户应用的`sepc`都会在被切换前记录在对应的“陷入”上下文中，因此应用恢复后会从上次被暂停的地方继续运行
       
       - 进程退出
         
         涉及进程退出时会先修改`PROCESSOR`中记录的当前进程的运行状态为僵尸态，然后将当前进程的子进程的所有父进程变更为初始进程并将子进程全部添加进初始进程的子进程。最后回收当前进程占用的内存。
         
         调用`schedule`函数查找下一个可运行的进程，后续流程与进程切换的流程一致。
    
    8. 操作系统退出
       
       操作系统的退出是判断当前退出的进程ID是`IDLE_PID`的时候调用`shutdown`，这个进程是第一个运行的用户程序`initproc`。因为`initpro`程序运行了`exec`，所以`initproc`进程被`user_shell`顶替，所以`IDLE_PID`指向的进程实际上是`user_shell`，当`user_shell`结束时操作系统退出。
